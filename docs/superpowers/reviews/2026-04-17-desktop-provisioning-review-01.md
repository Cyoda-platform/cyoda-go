## Significant issues
The Homebrew tap name reintroduces cyoda-go at the user's fingertips. Proposed tap repo is cyoda-platform/homebrew-cyoda-go, which makes the one-shot install brew install cyoda-platform/cyoda-go/cyoda. The whole motivation for Prerequisite A is user-facing cleanliness — but this invocation puts cyoda-go right next to cyoda in the command the README shows to new users. It also collides semantically with the main repo (cyoda-platform/cyoda-go), so a user reading the line reasonably wonders whether cyoda-go is the binary or the tap. Recommendation: use cyoda-platform/homebrew-cyoda as the tap repo, giving brew install cyoda-platform/cyoda/cyoda. Homebrew only requires the homebrew- prefix; the rest of the repo name is free.

macOS Homebrew users who skip cyoda init silently get the memory backend. Current plan puts the prompt in a caveats block, which Homebrew prints after install but which many users scroll past. When they later run cyoda, start writing data, restart, and find their data gone, they have no obvious reason why. Suggestion: post_install hook in the formula. The formula runs cyoda init automatically after install. cyoda init is already defined as idempotent (exits 0 if config exists), so re-installs and upgrades are fine. It makes Homebrew behave symmetrically with curl|sh, which is the right mental model.

install.sh fetched from main is a moving target. curl -fsSL .../main/scripts/install.sh | sh means whatever was last pushed to main is what new users run. That's standard practice (rustup does it) but it couples installer integrity to branch protection on main rather than to signed release artifacts. Two refinements worth considering: publish install.sh as a release asset so the URL is releases/latest/download/install.sh (same UX, tied to release artifacts), or at minimum document the trade-off and ensure branch protection + CODEOWNERS on scripts/install.sh. Not a blocker for v0.1.0; worth a sentence acknowledging it.

README version-pinning will rot. The install block hardcodes v0.1.0 in the .deb/.rpm URLs. Every release requires a README edit, which will drift. GitHub Releases supports /releases/latest/download/<filename> — pair it with a GoReleaser name_template that produces unversioned filenames (or symlink-style assets) and the README can stop moving. Alternative: document both an unversioned latest URL and a "to pin a specific version, use ..." pattern. Homebrew and go install are already version-agnostic in the listing; the .deb/.rpm rows are the only offenders.

Commented-out CYODA_SQLITE_PATH=<computed per-OS default> risks a silently-broken config. If the written file contains $XDG_DATA_HOME/cyoda/cyoda.db and the env-file parser doesn't expand shell variables, a user who uncomments the line gets a literal $XDG_DATA_HOME path. The spec doesn't say which. Two options: resolve the path at write time so the file contains an absolute path (simple, surfaces current value, no parser dependency), or make B2 explicit about variable expansion in env files. I'd pick write-time resolution — it's clearer to the user reading the file.

## Coordination cost worth naming
Prerequisite A landing as commits on shared PR #44 is more invasive than "a few renames." Shared PR #44 introduces .github/workflows/release.yml with cyoda-go as the binary/image name; those files have to land with post-rename strings to avoid a broken workflow on merge. Same for any test fixtures or script references inside #44. The coordination is feasible but the spec currently describes the rename as if it's a sprinkle of find-and-replace — in practice it's intertwined with #44's main content. Worth one sentence acknowledging that the rename and shared PR are co-dependent, not sequential.

Relatedly, Prerequisite A is motivated entirely from the desktop perspective but affects all three targets. The shared spec didn't anticipate this. That's fine — specs discover things — but it means either (a) the shared spec gets an addendum referencing this decision, or (b) the desktop spec formally supersedes the shared spec on artifact names. Either is clean; drifting with no acknowledgement isn't.

## Smaller observations
cyoda init presumes a subcommand dispatch pattern. From the current README, the binary appears to use flag-based invocation (cyoda-go --help, cyoda-go with no subcommands visible). Adding init means standing up a subcommand router — trivial with cobra, modest-but-nonzero with stdlib flag. Not a blocker, but worth acknowledging in the downstream plan so it doesn't surprise the implementer.

B1's Windows branch: Go's stdlib doesn't give you a data directory — os.UserCacheDir is cache-semantics (returns %LocalAppData% on Windows but ~/.cache on Linux), and os.UserConfigDir returns roaming %AppData% on Windows. The spec hedges with "via os.UserCacheDir analog... or direct %LocalAppData% env lookup." Pick one in the implementation and make the test matrix explicit — it's the kind of "obviously right" path that's actually not obvious in Go.

install.sh step 9 ("Invoke cyoda init") needs a failure mode. If cyoda init returns non-zero for some reason, does the installer fail, or does it print a warning and continue? I'd argue continue-with-warning — the binary is installed correctly, the config is just missing, and the user can re-run cyoda init themselves. Worth one line.

PAT vs GitHub App for the Homebrew tap automation. A fine-grained PAT scoped to one repo is the right choice for v0.1.0 — but it belongs to a person, so rotation and succession planning become a problem. A GitHub App is the modern answer. Not a blocker; worth a one-line "revisit if this project outlives the PAT owner" in the Out-of-Scope section.

brew install cyoda after tapping only works if no Homebrew core formula ships the name cyoda. Unlikely given the name's specificity, but worth knowing — if core ever adds a collision, users would need the long form again.

cyoda init exit code when config exists: the spec says exit 0 with a "already exists" message. That's consistent with "nothing went wrong," and it also happens to be what post_install block automation wants (idempotent, no failures on reinstall). Good call — worth noting explicitly that this is deliberate to support the post-install hook I suggested above.

The plugin-module tag coordination from Prerequisite B ("cut plugins/memory/v0.1.0, plugins/postgres/v0.1.0, plugins/sqlite/v0.1.0 fresh at the same commit") is exactly the sequencing dependency flagged in the shared-spec review. Nice to see it made concrete here.
